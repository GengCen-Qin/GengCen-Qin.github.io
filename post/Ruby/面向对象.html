面向对象

  1. 一个类，一个方法只做一件事儿
  2. 所有的实例数据，都通过统一访问器，进行统一访问，而不是堆散在代码的各个角落中
  3. 对于特定数据结构，可以使用Struct来封装。这样保证只有封装的地方需要知道数据结构的表达方式是什么。其他地方通过Struct对象，来获取信息即可。
  4. 推迟重构的心，不要急着将一些信息分离出去。可能未来这里永远的不会变。等到一天你需要变化时，再去重构也来得及
  5. 单一职责 -> 增加职责(但没有明确的目的去分离一个类出来)，那可以先试用Struct来做封装

管理依赖

class B
  def b_thing
  end
end

class A
  def build_something
    b = B.new
    a_thing * b.b_thing
  end
end

这里的A的build_something就有很多依赖

  1. 它知道B这个类
  2. 知道如何实例化B这个类
  3. 知道这个类有b_thing这个方法

这就导致A 和 B耦合

  1. 如果B要改名，A这个类的代码需要一起改
  2. 如果B的构造方法修改，A需要改
  3. 假如有个C，也想利用A.build_something，现在的代码无法直接复用，因为A 和 B绑定了

解决知道其他类，以及如何构造的依赖

  1. 依赖注入

也就是说，如果A依赖B，那在A实例化的时候，将B引入进来，作为实例变量

  1. 如果不依赖注入，也应该将类的构造隔离出去，也就是说单独写一个方法来获取构造好的外部类

解决知道其他对象的方法，以及如何调用他们的

  1. 隔离

比如 A.a_method 中有一行，执行了 b.b_method , 最好将这部分依赖单独放到一个方法中，这样如果b_method需要修改，那只单独修改该方法就好。a_method本身并不需要调整

  1. 定义函数参数时，尽量使用关键字参数，好处就是你不需要知道参数的顺序，并且你知道每个参数的含义
  2. 如果初始化方法中需要复杂代码构建信息，也应该放到一个方法中
  3. 依赖抽象而不是具体

创建灵活接口

  1. 面向类编程 -> 面向消息发送编程

以往我们会将注意力放在，有哪些类需要定义，每个类能做什么

现在我们应更多想想，我有哪些消息要发送，谁来发送，发送给谁

  1. 在依赖其他人时，我们更应该关注What，而不是How

也就是说你只需要告诉你依赖的人，你想要什么。至于具体怎么实现，并不需要你清楚。尽可能将信息封锁起来。

  1. 对他人知道的越少，依赖就越小
